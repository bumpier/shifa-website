3 Approaches to the Admin Panel

Pick the architecture that feels right before we design the details

A

New AdminUser model + role-based auth ★ Recommended

Add a new AdminUser table (email, passwordHash, role: ADMIN | PACKER). The existing env-var super-admin stays as a fallback escape hatch. Packers get their own login at /admin/login (shared URL, role detected after login). Admin dashboard shows full stats + charts; Packer dashboard shows orders only.

Pros

  • Audit trail per user
  • Easy to add more roles later
  • Minimal disruption to existing admin
  • Clean permission checks

Cons

  • New DB model + migration
  • Slightly more auth logic
B

Extend existing User model with admin roles

Add a PACKER role to the existing affiliate User model. Reuse the same auth system. Packers log in via the same /auth/login as affiliates, then get redirected to a restricted admin view.

Pros

  • No new model needed
  • Reuses existing auth code

Cons

  • Mixes affiliate + admin concerns in one table
  • Confusing UX (affiliate login = admin?)
  • Harder to lock down later
C

Full replacement — all admins become AdminUser records

Remove the env-var password entirely. Super-admin becomes the first AdminUser record seeded in the DB. All auth goes through the same table. Cleanest long-term, but requires a migration of the current admin login flow.

Pros

  • Cleanest architecture
  • No env-var escape hatch to maintain
  • Full audit trail for all admins

Cons

  • Bigger change, more risk
  • Breaks existing admin login during migration
  • No fallback if DB is down
